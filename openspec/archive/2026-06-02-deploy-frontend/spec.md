# Deploy frontend — Spec

## Capability groups

- **CDN** — S3 + CloudFront construct (10 requirements)
- **CACHE** — Cache behaviors + SPA fallback (6 requirements)
- **SSM** — SSM parameter publishing + API URL discovery (4 requirements)
- **BUILD** — Vite env injection at build time (3 requirements)
- **WORKFLOW** — `.github/workflows/deploy-frontend.yml` (8 requirements)
- **ROLLOUT** — One-time manual steps (3 requirements)

Total: **34 requirements / 8 scenarios**.

---

## CDN — S3 + CloudFront

### CDN-01

A new construct `WebDistribution` MUST live at `packages/infra-cdk/src/constructs/WebDistribution.ts`.

### CDN-02

The construct MUST create an S3 bucket with:

- `bucketName: 'smart-wallet-web-prod-{accountId}'` (suffixed to avoid global-namespace collisions)
- `blockPublicAccess: BLOCK_ALL`
- `removalPolicy: RETAIN` (never auto-delete on stack destroy)
- `autoDeleteObjects: false`
- `versioned: false` (cost — versions accumulate; we deploy fresh each time)

### CDN-03

The construct MUST create a CloudFront distribution targeting the bucket via **OAC** (Origin Access Control). NOT OAI (legacy).

### CDN-04

The distribution MUST set `priceClass: PRICE_CLASS_100` (US/Canada/Europe edges only).

### CDN-05

The distribution MUST set `viewerProtocolPolicy: REDIRECT_TO_HTTPS` on the default behavior.

### CDN-06

The distribution MUST set `defaultRootObject: 'index.html'` so requests to `/` serve `index.html`.

### CDN-07

The bucket policy MUST be auto-generated by CDK to allow `s3:GetObject` ONLY from the CloudFront distribution's OAC principal — direct S3 URLs MUST return 403.

### CDN-08

`SmartWalletStack` MUST instantiate `WebDistribution`.

### CDN-09

The bucket MUST be private — no `publicReadAccess: true`, no `cors: '*'`, no website hosting feature.

### CDN-10

The distribution and bucket MUST have `Tags.of(this).add('project', 'smart-wallet')` and `('component', 'web')` for cost tracking.

## CACHE — Cache behaviors + SPA fallback

### CACHE-01

The default cache behavior (`*` path pattern) MUST use `CachePolicy.CACHING_OPTIMIZED` (default TTL ~24h, max TTL 1 year) — optimal for hashed assets.

### CACHE-02

An ADDITIONAL behavior for path pattern `/index.html` MUST use `CachePolicy.CACHING_DISABLED` (TTL 0) so refreshed HTML is always served fresh.

### CACHE-03

The distribution MUST configure `errorResponses` for 403 and 404, both mapping to:

- `responsePagePath: '/index.html'`
- `responseHttpStatus: 200`
- `ttl: Duration.minutes(5)` (cached briefly to avoid hot-spotting S3 on 404s)

This makes React Router routes (e.g. `/dashboard`, `/recurring`) work on direct hit or refresh.

### CACHE-04

The distribution MUST allow the HTTP methods CloudFront's default set supports (`GET`, `HEAD`, `OPTIONS`).

### CACHE-05

The distribution MUST enable Gzip + Brotli compression (CDK default; no extra flag needed).

### CACHE-06

The construct MUST NOT enable CloudFront access logging (cost: extra S3 storage; out of scope per proposal §2).

## SSM — Parameter publishing + API URL discovery

### SSM-01

After creating the distribution, the construct MUST publish three SSM parameters:

- `/smart-wallet/prod/web/bucket-name` → bucket name
- `/smart-wallet/prod/web/distribution-id` → distribution id
- `/smart-wallet/prod/web/distribution-domain` → distribution `domainName` (no scheme prefix)

### SSM-02

A CfnOutput `WebDistributionUrl` with the full `https://{distributionDomain}` URL MUST be emitted, surfaced as the visible artifact of the CDK deploy.

### SSM-03

The API URL parameter at `/smart-wallet/prod/api/url` is NOT created by CDK. It is published one-time by the user as part of the rollout (manual `aws ssm put-parameter`). The workflow assumes it exists and reads it at deploy time.

### SSM-04

If `/smart-wallet/prod/api/url` is missing when the workflow runs, the SSM read step MUST fail loudly with a non-zero exit code, surfacing the gap to the user before the build step starts.

## BUILD — Vite env injection

### BUILD-01

Before `pnpm build` runs, the workflow MUST export these env vars from SSM values:

- `VITE_API_BASE_URL` ← `/smart-wallet/prod/api/url`
- `VITE_COGNITO_USER_POOL_ID` ← `/smart-wallet/prod/cognito/user-pool-id`
- `VITE_COGNITO_CLIENT_ID` ← `/smart-wallet/prod/cognito/user-pool-client-id`
- `VITE_COGNITO_REGION` ← `/smart-wallet/prod/region`

### BUILD-02

The workflow MUST build the web package specifically via `pnpm --filter @smart-wallet/web build` (NOT root `pnpm build`) so the deploy doesn't rebuild the api/domain/shared-types packages unnecessarily after they were already built in prior workflow runs. Build time difference is small but it keeps the deploy job focused.

Alternative accepted: full `pnpm build` is fine (turbo caches the upstream packages). The spec MAY use either form.

### BUILD-03

The workflow MUST NOT commit or persist the built artifacts back to the repo. The output `packages/web/dist/` is workflow-scoped only.

## WORKFLOW — `.github/workflows/deploy-frontend.yml`

### WORKFLOW-01

A file `.github/workflows/deploy-frontend.yml` MUST exist with `name: Deploy frontend`.

### WORKFLOW-02

Trigger: `on: push: branches: [main]`. NO `pull_request` (no preview deploys).

### WORKFLOW-03

Permissions: `id-token: write` + `contents: read` (OIDC + checkout).

### WORKFLOW-04

Concurrency: `group: deploy-frontend` with `cancel-in-progress: false` (S3 sync mid-flight should not abort).

### WORKFLOW-05

Steps in order: checkout, setup pnpm + Node, install (`--frozen-lockfile`), configure-aws-credentials (OIDC with `vars.AWS_DEPLOY_ROLE_ARN`), SSM read step, build step with env vars wired, sync step (`aws s3 sync packages/web/dist/ s3://${bucket} --delete`), invalidation step (`aws cloudfront create-invalidation --distribution-id ${id} --paths "/*"`).

### WORKFLOW-06

The build step MUST set `VITE_*` env vars from the SSM read step outputs.

### WORKFLOW-07

The sync step MUST use `--delete` so files removed from the build are also removed from S3 (otherwise stale assets accumulate).

### WORKFLOW-08

The invalidation step MUST NOT wait for invalidation completion (no `--wait` flag). Invalidation runs async; the workflow finishes faster, and the cache flush completes within ~3 min on AWS side.

## ROLLOUT — One-time manual steps

### ROLLOUT-01

The proposal §3 MUST include a "Rollout" section with the four ordered manual steps: merge PR → CDK deploy → put API URL SSM param → read distribution-domain SSM param.

### ROLLOUT-02

The proposal MUST document how to retrieve the current API URL if the SSM put step is needed (read from CloudFormation output of the `smart-wallet-api-prod` stack OR from `serverless info --stage prod`).

### ROLLOUT-03

The rollout MUST NOT require any change to the existing deploy-backend.yml workflow. The two workflows are independent.

## Scenarios

### S-01 — First CDK deploy provisions S3 + CloudFront

**Given** the construct does not exist yet in the deployed stack
**When** the user runs `cdk deploy` from local
**Then** CFN creates a new bucket, a new distribution with OAC, and writes the 3 SSM parameters. CfnOutput shows the new URL.

### S-02 — First workflow run after merge

**Given** the user merged the PR and configured the API URL SSM param
**When** GitHub triggers `Deploy frontend / deploy` on main push
**Then** the workflow assumes the OIDC role, reads 5 SSM values (api url + 3 cognito + 1 region), builds web with those env vars baked in, syncs to S3, invalidates CloudFront, finishes green within ~5 min.

### S-03 — Refreshing on /dashboard works

**Given** the app is deployed and the user is on `https://d3xyz.cloudfront.net/dashboard`
**When** they refresh the browser
**Then** CloudFront detects /dashboard is a 404 in S3 (only /index.html exists), returns /index.html with 200, React Router resolves the route client-side, the dashboard renders.

### S-04 — Asset cache busts on new deploy

**Given** the first deploy shipped `assets/index-abc.js`
**When** a second deploy ships `assets/index-xyz.js` (Vite generates a new hash)
**Then** the new `index.html` references the new hash. Old `index-abc.js` may still exist in S3 (will be cleaned by `--delete` if the file is gone) but no browser requests it because the HTML doesn't reference it.

### S-05 — SSM param missing

**Given** the user forgot to create `/smart-wallet/prod/api/url`
**When** the workflow runs the SSM read step
**Then** `aws ssm get-parameter` returns non-zero exit code, the step fails, the job stops before reaching the build. GitHub's job log shows the missing param name clearly.

### S-06 — Direct S3 URL is blocked

**Given** an external user finds the bucket name in CDK outputs
**When** they hit `https://smart-wallet-web-prod-XXX.s3.amazonaws.com/index.html`
**Then** S3 returns 403 Forbidden because the bucket policy only allows access from the CloudFront OAC principal.

### S-07 — Two deploys overlap

**Given** a deploy is in flight and a second push lands on main
**When** GitHub evaluates concurrency
**Then** the in-flight job continues to completion (`cancel-in-progress: false`). The new commit triggers a second job that queues until the first finishes.

### S-08 — Cache-bust is immediate for HTML

**Given** the user just deployed
**When** they refresh the page within seconds of deploy
**Then** CloudFront serves the new `/index.html` because the path-specific behavior has `CACHING_DISABLED`, so the request hits the S3 origin (which has the new file post-sync). Hashed assets continue serving from edge caches at original TTLs (they are immutable per hash).

## Glossary

- **OAC (Origin Access Control)**: AWS-recommended mechanism (2022+) to restrict S3 origins to CloudFront-only access. Modernizes the legacy OAI.
- **Cache behavior**: a CloudFront rule matching a path pattern to a cache policy, origin, and viewer protocol policy.
- **Price class**: a CloudFront pricing tier controlling which AWS edge locations serve content. `PRICE_CLASS_100` = US/Canada/Europe only — cheaper, sufficient for our user base.
- **Invalidation**: a request to CloudFront to remove specific paths from edge caches before their TTL expires. First 1000 paths/month are free; `/*` counts as one path.
- **SPA fallback**: redirecting 403/404 to `/index.html` so client-side routers (React Router) can resolve deep links. Without it, refreshing on `/dashboard` would 404 from S3.
